## Overview

This is the capstone phase. Every preceding phase built functionality; this phase builds confidence. Phase 69 is a systematic sweep across the entire KernSmith UI application to eliminate rough edges, enforce visual consistency, add first-run guidance, harden quality through testing, prepare release artifacts for all three platforms, and produce the marketing materials needed to launch.

Nothing in this phase adds new features. Instead, it transforms a functional application into a professional product that competes credibly with BMFont and Hiero on first impression, and surpasses them on sustained use.

The phase is organized into seven waves, each independently shippable but collectively required for a v1.0 release.

### Framework Stack

| Component | Library | Role |
|-----------|---------|------|
| **Rendering** | MonoGame (DesktopGL) | Cross-platform game loop, SpriteBatch rendering, texture management, input handling |
| **UI Layout** | GUM UI (code-only) | Declarative layout containers, text rendering, scroll viewers, buttons, text boxes — all constructed in C# code (no XAML, no .gumx files) |
| **Utilities** | MonoGame.Extended | Camera2D, input listeners, tweening/easing, sprite utilities |
| **Architecture** | MVVM (hand-rolled) | ViewModels use `INotifyPropertyChanged` and command pattern. No CommunityToolkit dependency — ViewModels are plain C# with zero MonoGame references |

### Interaction Patterns

- All animations respect the OS-level "reduce motion" accessibility setting (detected via platform API where available, otherwise via user preference). When reduced motion is requested, animations are replaced with instant transitions.
- Animations use MonoGame.Extended tweening or manual `GameTime`-based lerping. No `Thread.Sleep` or `Task.Delay` for animation timing.
- Empty states are driven by ViewModel state (e.g., `HasResult`, `HasFont` properties). The GUM UI layer checks these properties each frame and adjusts visibility/alpha accordingly.
- Maintain a consistent 60fps during all UI interactions. Profile with MonoGame's built-in `GameTime` tracking. Log frame drops in DEBUG builds.

### MonoGame-Specific Polish

| Item | Details |
|------|---------|
| **Frame rate consistency** | Verify 60fps during all UI interactions including animations. Use `IsFixedTimeStep = true` with `TargetElapsedTime = 1/60`. Profile with `GameTime.IsRunningSlowly` logging. |
| **GPU memory audit** | Audit all `Texture2D`, `RenderTarget2D`, and `SpriteFont` instances for proper disposal. Implement `IDisposable` on any screen/panel class that owns GPU resources. Verify no texture leaks during screen transitions using a DEBUG resource counter. |
| **Content pipeline vs runtime loading** | Finalize decisions: UI textures (icons, backgrounds) via Content Pipeline `.xnb` for optimal loading. User font textures and generated atlases via `Texture2D.FromStream` at runtime. Document the rationale. |
| **Window icon via SDL2** | MonoGame DesktopGL uses SDL2. Set the window icon using `SDL.SDL_SetWindowIcon` with a loaded surface if MonoGame's built-in icon API is insufficient. Wrap in platform-conditional code. |
| **SpriteBatch state management** | Audit all `SpriteBatch.Begin/End` pairs for correct blend state, sampler state, and depth stencil state. GUM UI rendering and custom MonoGame rendering must not interfere with each other's render state. |

## Notes

- This phase intentionally adds zero new features. Every task is about quality, consistency, documentation, or distribution. Feature creep is the primary risk.
- Wave 5 (Packaging) has the highest risk due to MonoGame native library bundling across platforms, code signing bureaucracy, and CI pipeline complexity. MonoGame DesktopGL bundles SDL2, OpenAL, and platform-specific native libraries that must all be present in the publish output. Start certificate procurement early.
- Since all UI layout is code-only (GUM, no XAML), the Wave 1 consistency audit is done entirely by grepping C# source files. This is simpler than auditing XAML but requires disciplined use of `UiTheme` and `UiConstants` classes.
- MonoGame does not provide native screen reader support. This is documented as a known limitation in Wave 4.15 and Wave 7.2. Keyboard navigation (Wave 1.10) partially mitigates this.
- All animations in Wave 2 must be `GameTime`-based (not wall-clock or `Thread.Sleep`). This ensures animations pause correctly if the window loses focus and scale correctly if frame rate drops.
- The manual test checklists (Wave 4) should be executed by someone other than the developer who built the feature being tested, if possible.
- All marketing screenshots (Wave 6) should be retaken after Wave 1 and Wave 2 are complete, not before — the UI will look different after the polish pass.
- The ship decision (7.15) is not a formality. If P1 issues are found, the release is delayed until they are resolved. Shipping broken software is worse than shipping late.
